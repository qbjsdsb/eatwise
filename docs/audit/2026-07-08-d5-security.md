**检查日期**：2026-07-08
**检查范围**：EatWise v0.33.0+46（分支 trae/agent-wX1X6Q）
**检查方法**：Grep 关键词扫描（debugPrint/print + apikey/token/secret/password、customSelect/customStatement/rawQuery、http://、硬编码密钥）+ 逐文件审查（secure_config_store / app_config / sentry_init / sentry_scrub / json_exporter / json_importer / auto_backup / connection / AndroidManifest / build.gradle.kts / AI providers / 更新模块）+ 对照 M23 既有审计（docs/audit/m23-dim4-security.md）核对历史发现闭环情况

## 检查项与结果

| # | 检查项 | 状态 | 说明 |

| 1 | API key / 密钥存储 | ⚠️ 部分通过 | SecureConfigStore 正确使用 flutter_secure_storage（iOS first_unlock + 禁 iCloud 同步，Android 默认 RSA OAEP + AES-GCM）；运行时无硬编码 key；但**文档中存在硬编码真实 GLM API key**（P0） |

## 发现的问题

### P0（严重）

#### P0-1：文档中硬编码真实 GLM API key（凭证泄露）

- **位置**：
  - `docs/superpowers/plans/2026-07-02-sprint2-full-loop.md:2105`
  - `docs/superpowers/plans/2026-07-02-sprint2-full-loop.md:2825`
  - `docs/superpowers/plans/2026-07-02-sprint2-full-loop.md:2907`
- **现状**：三处冒烟测试命令明文写入 `--dart-define=GLM_API_KEY=656dc50783a14252a2c49639b6323c31.U0xZQeRXxspLuYAS`。该值符合智谱 AI API key 格式（32 位 hex id + `.` + 16 位 secret），是真实可用凭证，已进入 git 历史。
- **影响**：任何能访问该仓库（含历史 commit）的人可提取此 key 调用智谱 AI API，产生费用或滥用。即便现在删除文件，git history 仍可追溯。
- **建议**：
  1. **立即到智谱开放平台吊销/轮换该 key**（最优先，修复代码无法撤销已泄露的凭证）；
  2. 将文档中的 key 替换为占位符 `--dart-define=GLM_API_KEY=你的key`；
  3. 若仓库可能公开（README/GitHub 主页同步功能已规划），考虑 `git filter-repo` 清洗历史（注意会改写 commit hash，需 force push）。
- **工作量**：吊销 5 分钟 + 文档替换 5 分钟（历史清洗可选）

### P1（高优先级）

#### P1-1：SQLite 数据库明文存储，与多处文档声称的"AES 加密"严重不符

- **位置**：`lib/data/database/connection.dart:10-19`
- **现状**：函数名仍为 `openEncryptedConnection`，但实现是 `NativeDatabase.createInBackground(dbFile)`，**未传入任何加密密钥/口令**。文件头注释自述"明文版，移除 sqlite3mc 加密避免 native 库兼容问题"。数据库文件 `eatwise.db` 明文落盘于应用沙箱目录。
- **数据暴露面**：明文 DB 含 `profiles`（age/gender/heightCm/weightKg/bodyFatPct/specialCondition/healthCondition/dietPreference）、`weight_logs`（体重/impedance/bodyFatPct）、`meal_logs`（全部饮食记录 + 图片路径）、`food_items`、`insight_summaries`（AI 健康汇总原文）。这是一份完整的个人健康画像。
- **文档-实现矛盾**（放大风险）：以下位置仍声称数据库加密，制造虚假安全感：
  - `README.md:44,148`（"AES 加密；密钥存 flutter_secure_storage"）
  - `pubspec.yaml:13`（"drift 2.32+ + sqlite3mc 加密"）
  - `.trae/specs/m25-github-homepage-sync/spec.md`、`docs/superpowers/specs/2026-07-01-eatwise-design.md`（多处"AES-256 加密"）
  - 项目规则 `.trae/rules/project_handoff.md` 未更正
  - 注：`docs/audit/2026-07-08-d2-database-migration.md:244-246` 已记录此矛盾，但本安全维度需独立标注其安全影响
- **影响**：root 设备 / 物理获取手机 / 应用沙箱被攻破时，全部健康隐私数据可直接用 sqlite3 命令读取；备份导出（P2-2）+ 明文 DB 形成双重暴露。
- **建议**：
  1. **优先**：同步更正所有声称"加密"的文档/规则/README，避免维护者误判安全基线；
  2. 将 `openEncryptedConnection` 重命名为 `openConnection`（消除误导性命名）；
  3. 评估恢复 sqlite3mc 加密（密钥派生自 flutter_secure_storage 随机串），或明确接受"个人自用 + 沙箱隔离 + allowBackup=false"为可接受风险并在文档声明。
- **工作量**：文档更正 30 分钟；重命名 10 分钟；恢复加密 4-6 小时（需验证 CI build hooks）

### P2（中低优先级）

#### P2-1：SecureConfigStore Android 配置注释不准确（文档问题，安全无影响）

- **位置**：`lib/core/config/secure_config_store.dart:32-34`
- **现状**：注释称"encryptedSharedPreferences 已废弃，默认即自动迁移到 custom ciphers"。实际上 EncryptedSharedPreferences 并未被废弃，flutter_secure_storage 默认 `useEncryptedSharedPreferences: false` 走的是库自带 RSA/ECB/OAEPwithSHA-256andMGF1Padding + AES/GCM 自定义实现，并非"自动迁移"。
- **影响**：安全层面无影响（默认 RSA OAEP + AES-GCM 已足够安全）；仅注释误导维护者。
- **建议**：修正注释为"Android 默认 RSA OAEP + AES-GCM 自定义实现；如需 Jetpack EncryptedSharedPreferences 可显式设 useEncryptedSharedPreferences: true"。
- **工作量**：5 分钟

#### P2-2：JsonExporter 导出全量健康隐私数据明文未加密

- **位置**：`lib/data/backup/json_exporter.dart:14-39, 48-69`
- **现状**：导出 profiles（含 bodyFatPct/healthCondition/dietPreference）、meal_logs（饮食记录 + 图片路径）、weight_logs（体重/impedance/bodyFatPct）、insight_summaries（AI 健康汇总）为明文 JSON。不含 API key/dsn（已规避）。
- **缓解因素**：自动备份存 `${applicationDocumentsDirectory}/backups/`（沙箱，外部应用不可访问）；`allowBackup=false` 防 ADB 备份。
- **影响**：root 设备可读；用户主动导出分享时无加密保护。
- **建议**：导出支持可选密码加密（AES-GCM 包装），或导出文件头加"含健康隐私数据，妥善保管"提示。
- **工作量**：2 小时

#### P2-3：AutoBackup 备份文件明文存储

- **位置**：`lib/data/backup/auto_backup.dart:18-29`
- **现状**：自动备份写明文 JSON 到沙箱 `backups/` 目录，保留最近 4 份。
- **影响**：与 P2-2 同源，root 设备可读历史 4 份健康快照。
- **建议**：与 P2-2 一并处理（加密方案统一）。
- **工作量**：随 P2-2

#### P2-4：无 certificate pinning

- **位置**：全项目网络层（AI providers 用 openai_dart 默认 dio/http；github_release_client / apk_downloader / off_provider 用 `package:http` 默认 Client）
- **现状**：所有外部请求走系统 CA 校验的 HTTPS，无证书锁定。
- **影响**：企业 CA / 被信任的恶意 CA 仍可中间人；个人自用 + 公网 API 场景风险低。
- **建议**：个人自用可接受现状；若分发他人可考虑对 AI API 域名加 pinning。
- **工作量**：4 小时

#### P2-5：apk_downloader 未校验下载 URL scheme + 无 APK 完整性校验

- **位置**：`lib/core/update/apk_downloader.dart:39-47`
- **现状**：`download(url:)` 接受任意 URL，未强制 https；URL 来自 GitHub `browser_download_url`（默认 HTTPS）但代码层无校验。下载后无 checksum/签名校验。
- **缓解因素**：TLS 保护传输；Android 包安装器校验 APK 签名（签名不一致弹窗拒绝）；keystore 不进 repo。
- **影响**：理论极低概率；若 GitHub release 被替换为同签名恶意包可绕过。
- **建议**：download() 入口加 `if (!url.startsWith('https://')) throw`；release 元数据加 SHA256 校验下载后比对。
- **工作量**：30 分钟

#### P2-6：设置页 Base URL 输入无 https 校验

- **位置**：`lib/features/settings/settings_page.dart`（Qwen/GLM Base URL 输入框）
- **现状**：用户可填任意 URL 含 `http://`，若误填则 API key 随 Authorization header 明文传输。
- **影响**：用户配置错误致 API key 网络明文暴露。
- **建议**：保存时校验 scheme == https，非 https 弹警告确认。
- **工作量**：30 分钟

#### P2-7：boot_log.txt 写入未脱敏的异常 + 堆栈

- **位置**：`lib/main.dart:25-34`
- **现状**：启动期异常 + stack 原样写入 `${applicationDocumentsDirectory}/boot_log.txt`，未用 sentry_scrub 同款正则脱敏，无大小上限滚动。
- **缓解因素**：沙箱目录，外部应用不可访问；release debugPrint 被裁剪但 boot_log 是落盘文件不受此影响；实际异常消息一般不含 key 值。
- **影响**：root 设备可读；理论若异常消息拼接了 key 值会落盘。
- **建议**：写入前复用 `_scrubString` 脱敏；限制最大 100KB 滚动覆盖。
- **工作量**：20 分钟

#### P2-8：AppConfig 保留 --dart-define fallback（key 可进 APK 二进制）

- **位置**：`lib/core/config/app_config.dart:53-62`
- **现状**：secure_storage 为空时 fallback 到 `String.fromEnvironment('QWEN_API_KEY'/'GLM_API_KEY'/'SENTRY_DSN')`。若用 `--dart-define=KEY=xxx` 构建，key 编译进 Dart snapshot，可被 `strings`/逆向提取。
- **缓解因素**：CI release.yml 构建命令 `flutter build apk --release` **不带** --dart-define，release APK 不含 key；此路径仅 dev 本地用。设置页引导用户用 secure_storage 而非 dart-define。
- **影响**：仅当开发者误用 `--dart-define` 构建分发包时泄露。
- **建议**：release 模式 + dart-define 值非空时打警告日志；或文档明确"仅 dev 用 dart-define"。
- **工作量**：30 分钟

#### P2-9：README/pubspec 仍声称数据库 AES 加密（文档-安全矛盾）

- **位置**：`README.md:44,148`、`pubspec.yaml:13`
- **现状**：与 P1-1 同源，文档声称 sqlite3mc AES 加密，实际明文。
- **影响**：维护者/用户误判安全基线，可能放松其他防护。
- **建议**：随 P1-1 一并更正。
- **工作量**：随 P1-1

## 改进建议

### 立即处置（P0/P1）

1. **吊销并轮换泄露的 GLM API key**（P0-1）——这是唯一无法靠改代码挽回的发现，凭证一旦泄露只能吊销。
2. 更正所有声称"数据库加密"的文档/README/规则（P1-1 / P2-9），消除虚假安全感。
3. 评估是否恢复 sqlite3mc 加密；若决定接受明文风险，则在文档明确声明并评估是否需要降低其他数据敏感度（如不存储 bodyFatPct 历史明文）。

### 短期加固（P2，按投入产出排序）

4. apk_downloader + settings_page URL https 校验（P2-5、P2-6，合计 1 小时，堵住 API key 明文传输路径）。
5. boot_log.txt 脱敏（P2-7，20 分钟）。
6. SecureConfigStore 注释更正（P2-1，5 分钟）。

### 中期优化（可选）

7. 备份/导出加密（P2-2、P2-3，2 小时）。
8. APK 下载 SHA256 校验（P2-5，30 分钟）。
9. certificate pinning（P2-4，仅分发他人时考虑）。

## 整体评价

EatWise 在密钥运行时存储（SecureConfigStore 规范，无 instance 误用）、SQL 注入防护（drift 全参数化）、敏感日志管控（无 print、无 key 明文打印）、Sentry PII 过滤（beforeSend 全面脱敏 + tags 已补）、Android 权限最小化（allowBackup=false + 权限各有必要用途）五方面**符合最佳实践**。

本次新增两个关键发现：(1) 文档历史中遗留的真实 GLM API key（P0，需立即吊销）；(2) SQLite 数据库实际明文存储但多处文档声称加密（P1，文档-实现矛盾放大风险）。两者均非运行时代码逻辑缺陷，而是**凭证卫生**与**文档-实现一致性**问题，修复成本低但安全收益高。修复 P0/P1 后，EatWise 的安全基线在个人自用 app 中可达**良好水平**。
