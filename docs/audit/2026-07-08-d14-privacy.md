# D14 隐私检查报告（个人自用简化版）

**检查日期**：2026-07-08
**检查范围**：EatWise v0.33.0+46（分支 trae/agent-wX1X6Q）
**检查性质**：个人自用简化版（不上架应用商店，无第三方用户，评估口径以"开发者本人作为唯一用户的知情权与数据可控性"为准）
**检查方法**：逐文件审查 AI provider 实现（qwen_vl / glm_4v / glm_flash / off_provider）+ Sentry 配置（sentry_init / sentry_scrub）+ 备份导出（json_exporter / json_importer / auto_backup / image_cleanup）+ 图片处理流程（recognize_controller）+ AI 推荐数据流（ai_recommendation_prompt / ai_recommendation_service）+ UI 文案（recognize_page / insight_page / settings_page / me_page）+ AndroidManifest 权限 + 网络请求头扫描（Grep `Authorization|api-key|Bearer` + `https?://` + `EXIF|metadata` + `云端|上传|同意|隐私`）+ 对照 `assets/privacy_policy.md` 验证声明与实现一致性，并与 D5 安全审计交叉核对避免重复

## 与 D5 安全维度的边界

D5 聚焦"凭证卫生 / SQL 注入 / 日志管控 / 证书 / Manifest 加固"等**安全防护**；D14 聚焦"用户对数据流向的**知情度与可控度**"。两者在"备份明文未加密""Base URL 无 https 校验""boot_log 未脱敏"等项有交集，D14 仅从隐私视角复述并标注交叉引用，不重复展开技术细节。

## 发现的问题

### P0（隐私严重泄露）

无。

说明：生产代码无明文 HTTP 传输；图片 EXIF 默认剥离（防 GPS 泄露）；Sentry 默认关闭且脱敏；无硬编码用户身份信息外发；无第三方追踪 SDK。D5 发现的"文档中硬编码 GLM API key"（P0-1）属凭证泄露范畴，不纳入 D14 隐私维度。

### P1（用户不知情的数据外传）

#### P1-1：拍照识别 UI 无"图片将发送到云端"提示

- **位置**：`lib/features/recognize/recognize_page.dart:305-315`
- **现状**：拍照识别页 hero 区文案为"拍照识别食物"+"AI 自动识别菜品 + 估算营养，也可选相册图片"。两个按钮"拍照识别""从相册选择"均无任何关于"照片将上传到阿里云百炼/智谱 AI"的提示。用户点按钮即触发图片压缩 + base64 + 上传 Qwen-VL（失败降级 GLM-4V-Plus），全程无即时知情提示。
- **政策对照**：`assets/privacy_policy.md:13-16` 声明"拍照识别功能会将压缩后的食物照片（已剥离 EXIF 元数据）发送到以下大模型厂商"，实现与政策一致（EXIF 确实剥离、确实发送到 Qwen/GLM），但**用户不主动点开政策就不知道照片要外发**。
- **影响**：用户首次使用拍照识别时，对"照片离开本机"无即时感知。个人自用场景下用户即开发者本人，理论上知情，但 UI 透明度不足违反"数据外发应在动作发生处即时提示"的隐私设计原则。
- **建议**：在拍照识别页 hero 副标题下方加一行小字"照片将上传至 AI 云端识别（可查看隐私政策）"，或在首次拍照时弹一次性提示"图片将发送到 Qwen/GLM 云端识别，是否继续？"。
- **工作量**：15 分钟（文案）/ 30 分钟（首次弹窗 + 偏好记录）

#### P1-2：AI 推荐/汇总 UI 无"数据将发送到智谱 AI"提示

- **位置**：
  - AI 汇总：`lib/features/insight/insight_page.dart:371-414`（_generate 方法直接调 GLM-4-Flash）
  - AI 推荐：`lib/nutrition/ai_recommendation_service.dart:174-235`（_fetchFromAi 聚合上下文调 GLM-4-Flash）
- **现状**：
  - AI 汇总按钮点击后，将近 7/30 天的 `daily_calories` / `daily_weights` / 宏量营养素 / 饮食偏好食物 / 餐次分布 / streak / 体重变化 / **特殊人群画像（孕期/哺乳期/老年/糖尿病/高血压/肾病等）** 发送到智谱 GLM-4-Flash（见 `glm_flash_provider.dart:35-57` 的 `_buildPrompt` 与 `_appendEnhancedInsights`）。
  - AI 推荐发送的上下文更完整：profile 全字段（性别/年龄/身高/体重/体脂率/活动量/目标/特殊状况/健康状况/饮食偏好/每日热量目标）+ 当日剩余额度 + 近 14 天饮食历史 top 20 + 近 30 条推荐反馈（见 `ai_recommendation_prompt.dart:92-115` 的 `buildUserPrompt`）。
  - 两个功能页面的 UI 按钮均无"数据将发送到智谱 AI 云端"提示。
- **政策对照**：`assets/privacy_policy.md:18` 声明"AI 汇总建议功能会将近 7 天的热量与体重统计数据（不含照片）发送到智谱 AI"，但**未声明 AI 推荐功能**（政策只提了汇总），且**未声明发送特殊人群健康状况**（孕期/糖尿病/高血压/肾病属敏感健康信息）。
- **影响**：
  1. AI 推荐功能未在隐私政策中列出（政策-实现不一致）；
  2. 特殊人群健康状况（如糖尿病/高血压/孕期）作为敏感健康信息外发，用户无即时感知；
  3. AI 推荐发送的是"完整健康画像 + 饮食历史"，敏感度高于 AI 汇总（汇总只发聚合统计），但透明度反而更低。
- **建议**：
  1. 隐私政策补充 AI 推荐功能的数据流声明，明确列出"发送 profile 全字段 + 近 14 天饮食历史 + 反馈"；
  2. AI 推荐首次触发时弹一次性提示，告知将发送的健康数据范围（尤其特殊人群健康状况）；
  3. 考虑让特殊人群字段（specialCondition/healthCondition）在发送前由用户勾选是否包含。
- **工作量**：政策补充 15 分钟；首次弹窗 30 分钟；字段勾选 1 小时（可选）

### P2（建议增强透明度）

#### P2-1：隐私政策未列出 GitHub 更新检查（暴露 IP + User-Agent）

- **位置**：`lib/core/update/github_release_client.dart:31-47`
- **现状**：应用更新检查调 `https://api.github.com/repos/qbjsdsb/eatwise/releases/latest`，发送 `Accept: application/vnd.github+json` + `User-Agent: EatWise-Updater` 头。GitHub 会记录请求方 IP（匿名访问仍暴露 IP）。隐私政策未声明此数据流。
- **影响**：用户不知情 App 启动/手动检查更新时会向 GitHub 发起请求暴露 IP。
- **建议**：隐私政策"第三方数据传输"章节补充"应用更新检查会访问 GitHub API（暴露 IP 地址，匿名访问无身份信息）"。
- **工作量**：5 分钟

#### P2-2：隐私政策未列出 Open Food Facts 菜名查询

- **位置**：`lib/ai/off_provider.dart:40-107`
- **现状**：本地食物库未命中时，OffProvider 调 `https://world.openfoodfacts.org/cgi/search.pl` 查品牌食品营养，发送菜名文本 + User-Agent `EatWise/{version} (Android; food-matching) contact@eatwise.app`。隐私政策未单独列出 OFF 数据流（政策说"仅发送识别所需的图片/数据"过于笼统）。
- **缓解因素**：OFF 只传菜名不传图片/身份，敏感度低；且 OFF 是兜底数据源（本地库命中则不调）。
- **建议**：政策补充"本地食物库未命中时，菜名会发送到 Open Food Facts 查询营养数据（不传图片、不传身份）"。
- **工作量**：5 分钟

#### P2-3：隐私政策未声明 boot_log.txt 本地异常日志

- **位置**：`lib/main.dart:25-34`
- **现状**：启动期异常 + 堆栈原样写入 `${applicationDocumentsDirectory}/boot_log.txt`，未脱敏，无大小上限滚动。隐私政策"数据存储"章节未提及此本地日志文件。
- **交叉引用**：D5 P2-7 已记录安全视角，本项从隐私视角复述——异常消息理论可能拼接用户数据（如 AI 返回的菜名含在 exception message 中），落盘后未脱敏。
- **建议**：政策补充"启动异常会写入本地 boot_log.txt（沙箱内，不外发）"；并复用 sentry_scrub 的 `_scrubString` 脱敏（与 D5 建议一致）。
- **工作量**：政策 5 分钟 + 脱敏 20 分钟

#### P2-4：Sentry 脱敏词表未覆盖本 app 特有 PII 字段名

- **位置**：`lib/core/error/sentry_scrub.dart:78-83`
- **现状**：`_isSensitiveKey` 敏感词列表覆盖 food/dish/serving/weight/calorie/kcal/protein/fat/carb/api_key/key/token/secret/password/dsn/image_path/thumbnail/original_image，但**未覆盖** `health`/`diet`/`special_condition`/`impedance`/`body_fat`/`gender`/`age`/`height` 等本 app 特有的健康 PII 字段名。
- **影响**：若异常 extra/tags 的 key 恰好是 `health_condition` / `special_condition` / `body_fat_pct` / `impedance`，因其不含 `health`/`diet` 子串（`health_condition` 含 `health`，`special_condition` 不含任何敏感词，`body_fat_pct` 含 `fat` 会被命中，`impedance` 不含任何敏感词），会保留 key 名（value 仍会经 `_scrubValue` 处理）。实际风险低（key 名本身非 PII 值，value 经 `_scrubString` 脱敏路径/key 模式）。
- **缓解因素**：代码中无 `Sentry.setTag`/`setUser` 主动注入这些字段；exception message 多是技术错误不含健康值；`request` 整体置 null 防止 HTTP body 泄露。
- **建议**：敏感词列表补充 `health`/`diet`/`special`/`impedance`/`gender`/`age`/`height`（防御性，覆盖本 app PII 字段名）。
- **工作量**：5 分钟

#### P2-5：备份导出明文未加密（与 D5 P2-2/P2-3 同源，隐私视角复述）

- **位置**：`lib/data/backup/json_exporter.dart:14-39`、`lib/data/backup/auto_backup.dart:18-29`
- **现状**：导出 7 张表明文 JSON，含 profiles（bodyFatPct/healthCondition/dietPreference）、weight_logs（impedance/bodyFatPct）、meal_logs（饮食记录 + 图片路径）、insight_summaries（AI 健康汇总原文）。自动备份存沙箱 `backups/` 保留 4 份，手动导出存沙箱根目录。
- **隐私视角**：导出文件是"完整健康画像副本"，用户分享导出文件（如发微信/邮件）时无加密保护，接收方可见全部健康隐私数据。
- **交叉引用**：D5 P2-2/P2-3 已记录安全视角，建议方案一致（可选密码加密 AES-GCM 包装 / 文件头提示"含健康隐私数据妥善保管"）。
- **建议**：导出按钮下方加提示"导出文件含完整健康数据，请妥善保管勿随意分享"；中期支持可选密码加密。
- **工作量**：提示 5 分钟 / 加密 2 小时

#### P2-6：无数据库历史记录自动清理策略

- **位置**：`lib/data/backup/image_cleanup.dart`（仅清理图片，不清理 DB 记录）
- **现状**：ImageCleanup 仅删除超期原图文件 + 清除 DB 路径引用，**不删除 meal_logs/weight_logs/insight_summaries 记录本身**。数据库记录永久累积。
- **影响**：
  1. 长期使用 DB 膨胀（meal_logs 含 componentsSnapshotJson 可达数 KB/条）；
  2. insight_summaries 永久保留 AI 生成的健康汇总原文（含特殊人群画像）；
  3. 即使用户选"7 天保留期"删了原图，饮食记录文本仍永久留存。
- **缓解因素**：个人自用数据量有限；DB 在沙箱内；allowBackup=false 防 ADB。
- **建议**：设置页加"清理 N 天前饮食记录"选项（删 meal_logs + 级联删 recognition_feedbacks + 清理 insight_summaries），与图片保留期解耦。
- **工作量**：2 小时

#### P2-7：AI 推荐发送敏感健康状况无单独提示（与 P1-2 关联）

- **位置**：`lib/nutrition/ai_recommendation_prompt.dart:117-140`（_profileSection）
- **现状**：AI 推荐的 user prompt 含"特殊状况：孕期/哺乳期/老年/青少年"+"健康状况：糖尿病/高血压/肾病"+"饮食偏好：蛋奶素/纯素/乳糖不耐/无麸质"等敏感健康信息，发送到智谱 GLM-4-Flash。
- **隐私视角**：糖尿病/高血压/孕期属"特殊类别个人数据"（GDPR 第 9 条 / 个人信息保护法第 28 条定义的敏感个人信息），即便个人自用，外发到第三方 AI 厂商应有更高透明度。
- **建议**：与 P1-2 一并处理，在 AI 推荐首次触发提示中明确"将发送健康状况（如糖尿病/高血压等）"；或提供"不发送健康状况"开关让用户选择是否包含。
- **工作量**：随 P1-2

#### P2-8：OFF User-Agent 含虚假联系邮箱

- **位置**：`lib/ai/off_provider.dart:90-91`
- **现状**：User-Agent 为 `EatWise/{version} (Android; food-matching) contact@eatwise.app`。`contact@eatwise.app` 域名未在项目其他处出现，疑为占位邮箱（项目无对应域名资产）。
- **影响**：无隐私泄露（邮箱不对应真实用户），但 OFF 社区可能据此邮箱联系反馈，邮箱无效会致反馈流失。
- **建议**：改为真实可达邮箱，或移除 contact 部分（OFF 仅要求 UA 标识应用，不强制邮箱）。
- **工作量**：2 分钟

## 整体评价

EatWise 在隐私保护的基础设施层面**符合个人自用 app 的良好实践**：

- **EXIF 默认剥离**（`recognize_controller.dart:208` keepExif 默认 false）防止照片 GPS/设备信息泄露，与隐私政策声明一致；
- **Sentry 默认关闭**（`SecureConfigStore.getSentryEnabled` 默认返回 false）+ 全面脱敏 + 可关闭，用户对错误上报有完全控制权；
- **OFF 只传菜名不传图片/身份**（`off_provider.dart:35` 注释明示隐私设计）；
- **图片有清理策略且用户可配**（7/30/永久，设置页下拉）；
- **API key 以 Authorization Bearer 头发送**（openai_dart SDK 标准方式），生产代码全 HTTPS；
- **导入有二次确认弹窗**（`backup_page.dart:162-180` 知情同意设计）。

主要不足在**数据外发的即时透明度**：拍照识别与 AI 推荐两个核心功能在 UI 动作发生处无"数据将发送到云端"提示，用户需主动点开隐私政策才能知晓数据流向；AI 推荐功能本身未在隐私政策中列出，且发送的特殊人群健康状况（糖尿病/高血压/孕期）敏感度高于政策声明的"热量与体重统计"。修复成本低（P1 合计约 1 小时），收益是"用户在每次数据外发时都有即时感知"。

与 D5 安全维度的交叉项（备份明文 / Base URL 无 https 校验 / boot_log 未脱敏 / 无 certificate pinning）状态未变，详见 D5 报告，本报告不重复展开技术细节。修复 P1 + P2 短期项后，EatWise 的隐私透明度在个人自用 app 中可达**良好水平**。
