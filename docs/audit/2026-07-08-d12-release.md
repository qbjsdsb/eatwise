### P1（发布质量）

#### P1-1：v0.33.0 release 缺少 3 个 P1 bug 修复（tag 时机过早）

- **位置**：git history
- **现状**：
  - `v0.33.0` tag 指向 `bb30873`（`fix(build): 修复 build_runner 失败`）
  - tag 之后有 2 个 commit 未包含在 release 中：

    - `b140745 feat: 了解项目进展`（HANDOFF 更新，无代码影响）
  - `d36b3b0` 修复内容（来自 commit message）：

- **影响**：v0.33.0 release APK 缺少上述 3 个 P1 修复。用户安装 v0.33.0 会遇到：(1) recognize_page 写库中途失败留下脏数据；(2) 切换 BMR 公式后目标热量不更新；(3) v1 体脂秤用户升级后无法捕获体重（v0.31.0 功能回归）

  2. 或：若 v0.33.0 尚未广泛安装，删除 tag 重新打在 `d36b3b0` 之后（注意 GitHub Release 也需更新）；
  3. 长期：发版前在 release.yml 增加 "检查 working tree clean + HEAD 包含所有预期 commit" 步骤
- **工作量**：发 v0.33.1 约 10 分钟（bump + tag + push）

#### P1-2：release.yml 不跑 flutter analyze / flutter test（发版无质量门禁）

- **位置**：`.github/workflows/release.yml:45-83`
- **现状**：release.yml 构建步骤序列为：`pub get` → `build_runner` → `注入 keystore` → `build release APK` → `build debug APK`。**没有 `flutter analyze` 和 `flutter test` 步骤**。
- **影响**：发版时无静态分析 + 无测试回归校验。若开发者在本地跳过测试直接打 tag，CI 会照常构建发版，analyer warning / 测试失败无法拦截。对比 ci.yml 有完整的 analyze + test 步骤，但 ci.yml 不在 tag 触发链路上。
- **建议**：在 release.yml 的 `pub get` 之后、`build release APK` 之前插入：
  ```yaml
  - name: Analyze
    run: flutter analyze --no-fatal-infos
  - name: Test
    run: flutter test --exclude-tags smoke
  ```
  失败则阻断发版（GitHub Actions 默认非零退出码阻断后续 step）
- **工作量**：5 分钟

#### P1-3：ci.yml 只对 main 触发，发版分支从未经 CI（origin/main 停在 v0.25.0）

- **位置**：`.github/workflows/ci.yml:3-7`
- **现状**：
  - ci.yml 触发条件：`push: branches: [main]` + `pull_request: branches: [main]`
  - `origin/main` 最新 commit 为 `a27b347`（v0.25.0 时期，2026-07 发版提交）
  - 所有开发在 `trae/agent-wX1X6Q` 分支进行，**从未合并到 main**
  - 后果：v0.26.0 ~ v0.33.0 共 8 个版本的发版分支**从未经过 CI 校验**（analyze/test/build_runner 一致性）
- **影响**：发版质量完全依赖开发者本地自觉跑 analyze/test。若本地环境与 CI 不一致（如 Flutter 版本差异、生成产物漂移），问题无法在发版前发现。
- **建议**：
  1. 短期：将 ci.yml 触发条件扩展为 `branches: [main, 'trae/agent-*']`，或改为 `branches: ['**']` 全分支触发；
  2. 中期：将 `trae/agent-wX1X6Q` 合并到 main，同步开发与发版主干；
  3. 长期：规范分支策略（main 为发版主干，feature 分支 PR 合并）
- **工作量**：改触发条件 2 分钟；合并 main 需评估冲突

#### P1-4：ci.yml 与 release.yml Flutter 版本不一致（stable vs 3.44.4）

- **位置**：
  - `.github/workflows/ci.yml:16-19`：`channel: stable`（不锁版本，拿最新 stable）
  - `.github/workflows/release.yml:33-37`：`flutter-version: '3.44.4'` + `channel: stable`（锁定 3.44.4）
- **现状**：CI 用最新 stable（当前可能已到 3.45+），发版用 3.44.4。
- **影响**：
  1. CI 通过 ≠ 发版能通过：若 stable 升级引入 breaking change，CI 在新版本上过但发版用旧版本可能行为不同；
  2. build_runner 生成产物可能与 Flutter 版本相关，CI 用新版本生成的产物与发版 3.44.4 生成的不一致，触发 ci.yml 的 "生成产物一致性校验" 误报；
  3. 测试基线漂移：CI 在新版本跑的测试结果不能代表发版实际质量
- **建议**：ci.yml 也锁定 `flutter-version: '3.44.4'`，与 release.yml 一致；或在 release.yml 注释说明为何锁版本
- **工作量**：2 分钟

### P2（改进建议）

#### P2-1：无回滚机制文档

- **现状**：release.yml 无回滚 step；项目无回滚 SOP 文档
- **影响**：发版后发现严重 bug，需手动操作（删 tag + 删 release + 重新打旧 tag），容易出错
- **建议**：在 `docs/` 下补回滚 SOP：(1) GitHub Release 标记为 pre-release（API 或 UI）；(2) 通知用户回退到上一版；(3) 修复后发新版本而非删除 tag（tag 不可变原则）
- **工作量**：写文档 15 分钟

#### P2-2：release.yml `make_latest: true` 固定，无法标记 pre-release

- **位置**：`.github/workflows/release.yml:214`
- **现状**：`make_latest: true` 硬编码，每次发版都标记为 latest
- **影响**：无法发 pre-release 版本（如 beta / rc），所有 tag 触发的构建都成 latest release
- **建议**：改为按 tag 后缀判断（如 `v*-*` 为 pre-release）：
  ```yaml
  prerelease: ${{ contains(steps.tag.outputs.release_tag, '-') }}
  make_latest: ${{ !contains(steps.tag.outputs.release_tag, '-') }}
  ```
- **工作量**：5 分钟

#### P2-3：release.yml 不校验 build_runner 生成产物一致性

- **位置**：`.github/workflows/release.yml:48-49`
- **现状**：release.yml 跑 `dart run build_runner build --delete-conflicting-outputs`，但不检查生成产物是否与提交的一致（ci.yml 有此校验，见 `ci.yml:27-35`）
- **影响**：若开发者提交的 `database.g.dart` 与 schema 不同步，release.yml 会重新生成并用于构建，但提交的产物仍是旧的——发版构建与仓库状态不一致
- **建议**：在 release.yml build_runner 后加与 ci.yml 相同的 `git diff --exit-code` 校验
- **工作量**：3 分钟

#### P2-4：无 Gradle 缓存（可加速构建）

- **现状**：release.yml 用 `subosito/flutter-action@v2` 的 cache（Flutter SDK + pub），但无 Gradle 缓存
- **影响**：每次构建重新下载 Gradle 依赖（约 1-2 分钟），30 分钟 timeout 下非关键但可优化
- **建议**：加 `actions/cache@v4` 缓存 `~/.gradle/caches` + `~/.gradle/wrapper`，或用 `gradle/actions/setup-gradle@v3`
- **工作量**：10 分钟

#### P2-5：keystore 未配置时静默回退 debug 签名（应更显著警告）

- **位置**：`.github/workflows/release.yml:51-71`
- **现状**：keystore 注入 step 中，`ANDROID_KEYSTORE_BASE64` 为空时 `exit 0`（不阻断），仅 `echo "⚠️ 未配置..."`。构建继续用 debug 签名。
- **影响**：若 GitHub Secrets 意外丢失/未配置，CI 仍会构建发版，但产出的 APK 用 debug 签名——**无法覆盖安装旧版**（签名不一致），用户需卸载重装丢数据。当前 release notes 已说明此情况，但开发者可能忽略 warning。
- **建议**：
  1. 短期：将 warning 改为 `::warning::` GitHub annotation（更显眼）；
  2. 中期：加 step 输出 `steps.build.outcome` + 在 Create Release 前校验签名（用 `apksigner verify` 检查 APK 签名类型，debug 则阻断）；
  3. 或：对 tag 触发（非 workflow_dispatch）的构建，keystore 未配置直接 fail
- **工作量**：加 annotation 2 分钟；加 apksigner 校验 20 分钟

#### P2-6：Force update release body step 用 heredoc 缩进 hack（脆弱）

- **位置**：`.github/workflows/release.yml:216-285`
- **现状**：因 `softprops/action-gh-release@v2` 在 release 已存在时可能不更新 body，用 curl PATCH 强制更新。body 用 python heredoc 构造，靠 `line[10:]` 去掉 10 空格缩进——依赖固定缩进，脆弱。
- **影响**：若后续维护者调整缩进，body 会残留前导空格或截断
- **建议**：将 body 模板抽到独立文件（如 `.github/release-body.md.tmpl`），用 envsubst 替换变量，避免 heredoc 缩进问题
- **工作量**：20 分钟

## 结论

EatWise 项目 v0.33.0 的**构建配置与硬约束全部合规**（R8 禁用 / abiFilters arm64 / 签名从 Secrets 注入 / keystore 不入库），APK 可正常构建安装，无 P0 发布阻断问题。但发布工程流程存在 **6 项 P1 发布质量问题**和 **6 项 P2 改进建议**，核心风险集中在：

1. **发版质量门禁缺失**（P1-2/P1-3）：release.yml 不跑 analyze/test，ci.yml 只对 main 触发但 main 停在 v0.25.0——v0.26.0~v0.33.0 共 8 个版本从未经 CI 校验，发版质量完全依赖开发者本地自觉。
2. **版本号管理不一致**（P1-1/P1-5/P1-6）：v0.32.0 tag 缺失跳过发版；v0.33.0 tag 打在 P1 修复之前（缺 3 个 bug 修复）；tag 指向的 commit 不在 origin/main 上。

**建议发版前优先处理**：P1-2（release.yml 加 analyze/test）+ P1-1（发 v0.33.1 含 d36b3b0 修复），这两项是阻断"发版即带 bug"的最小成本修复。其余 P1 项可逐步推进。
